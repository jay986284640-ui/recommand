# 功能规格说明:优惠推荐 Agent (Promo Recommend Agent)

**功能分支**: `001-promo-recommend-agent`
**创建日期**: 2026-06-12
**状态**: 草稿
**输入描述**: "设计一个优惠内容推荐的 agent，使用主 agent+子 agent 的架构，第一期只有优惠推荐的功能，后续可能会扩展积分商城相关的能力。给出主 agent，子 agent 的功能划分，交互方式。"

## 范围与架构概述

本功能交付一套 **多 Agent 推荐系统**:H5(移动 H5 页面)用户用自然语言描述自己的需求,系统返回个性化的本地优惠内容(美团门店券、自拓展门店券、本地优惠买单、外部券),并能在对话中继续完成下单、改单、支付(走 H5 收银台)、查询自己券包(可用 / 已用 / 过期 / 详情 / 横向对比)的全流程;**第一期同时支持**:对话式排除与跨会话偏好记忆(显式 + 隐式,隐式默认关闭)、联想+语义召回(候选不足时)、按距离/热度/价格/推荐模型四种排序键、长会话的上下文压缩(主 Agent 自有能力)、失败交易的会话回溯与离线归因(供运营优化)。

**架构形态**:在最前面是**一个主 Agent(路由/编排者)**,它对接多个**子 Agent**,每个子 Agent 各自负责一个垂直能力。第一期只交付**一个子 Agent——本地优惠子 Agent**,它同时承担"发现推荐"与"下单/改单/支付"两类工具(都属于"优惠生命周期"这一垂直能力);后续阶段(第二期)再以"纯新增"的方式接入**积分商城子 Agent**(或其他子 Agent),无需修改主 Agent 的对外接口。

**领域上下文(为什么要拆成两个垂直,而非一个统一 Agent)**:
- **本地优惠**与**积分商城**所售物品无相关性,功能形态类似但**实现层无可复用点**:
  - **本地优惠**的内容形态:美团门店优惠券、自拓展门店优惠券、本地优惠买单、外部券。**支付模型是收银台**(用户下单时走标准收银台支付流程)。
  - **积分商城**的内容形态:积分商城实物/虚拟商品。**支付模型是积分扣减**(消耗用户既有积分,无收银台接入)。
- 两者的库存系统、优惠规则、资格校验、支付/兑换链路均独立;**没有可共享的业务组件**,只共享"Envelope 契约 + 路由层"。
- 这一点强化了"按垂直划分子 Agent"的架构选择——它们本就是两个独立产品,只是都被嵌在同一个 H5 入口下。

**为什么采用"主+子"分层,而不是单一 Agent**:
- 第一期只做一个垂直(优惠),如果把它做成单一 Agent,工具集与提示词会迅速膨胀,工具选择会变得不可靠。
- 主 Agent 是唯一持有"用户面对面对话状态"的角色,后续可以做到跨垂直交接(例如"用积分换这个券"→ 积分商城 + 本地优惠),而不会让状态泄漏到任何一个子 Agent。
- 子 Agent 之间可以独立版本化、独立扩容,适应后续商品库与垂直能力增长。
- **为什么不下单也单独拆子 Agent**:第一期只有一个垂直需要下单,独立子 Agent 会带来"多一跳 + 序列化跨 Agent 传上下文"的复杂度,但没有真实收益(Constitution V·Simplicity / YAGNI)。所以下单工具下沉到本地优惠子 Agent 内部;第二期积分商城自带的"兑换下单"是另一套完全不同的链路(积分扣减 ≠ 收银台),与本地优惠的下单工具**没有可复用点**,届时积分商城子 Agent 会自带自己的下单/兑换工具,不需要回到第一期的下单工具。

## 澄清记录

### Session 2026-06-12

- Q: 同一 user 在多个设备(手机 + PC)同时打开 H5 对话时,系统应如何处理? → A: **同 user 共享同一份会话状态**。H5 在多端共享同一份 session state(当前列表、草稿订单、排序键、排除项等);排序、排除、改单 / 下单操作在多端同步刷新;**支付前主 Agent 必须做草稿互斥校验**,避免重复下单。
- Q: 当外部服务(ES、推荐服务)不可用时,LP 子 Agent 应如何降级? → A: **按"降级阶梯"差异化处理**。ES 异常 → 用最近的缓存候选;推荐服务异常 → 跳过 LLM 排序,直接用热度榜;两路全挂 → 友好错误 + 重试入口。具体降级路径在 FR-012b / FR-014b 中锁定。
- Q: 当用户要求"删除我的数据"时,系统应如何处理(偏好 / 画像 / 回溯记录 / 草稿订单等)? → A: **显式偏好支持"立即清除";隐式偏好自动过期;回溯记录按聚合 + 脱敏保留 N 个月(可分析但不可定位个人)**。具体清除 / 保留规则在 FR-019b 中锁定。
- Q: 如何处理 per-user 的限流 / 防刷? → A: **per-user 限流 + 明确提示**;每登录用户每分钟 ≤ 50 次输入(明确回答:50 次/分钟),下单相关操作可单独更严格(由 plan 阶段细化);超限返回明确提示,绝不静默丢消息。具体规则在 FR-055 中锁定。
- Q: 新用户(无画像 / 无行为序列)首轮推荐应如何处理? → A: **走"全站热度榜"但不带任何角标**,用户体感与老用户一致;具体行为在 A-009 与 FR-014c 中锁定。

### 用户故事 1 —— 对话式本地优惠发现(优先级:P1)

用户在 H5 页面输入或说出类似 *"我想喝咖啡,附近有什么优惠"* 或 *"有没有适合 2-3 人聚餐的门店"* 的需求,期望得到一份个性化的优惠内容列表,并能在此基础上继续对话式地微调(如"再便宜一点的"、"距离我 3km 以内")。

**本垂直支持的内容形态**(由用户澄清的事实):
- **美团门店优惠券**——平台已接入的美团门店发行的券
- **自拓展门店优惠券**——客户自运营门店发行的券
- **本地优惠买单**——可优惠买单的门店(用户到店出示买单)
- **外部券**——来自其他渠道/合作方的券

这四类内容可以出现在同一张推荐列表中;**下单/支付链路在内部统一为收银台**(用户最终走 H5 收银台支付流程)。

**为什么是这个优先级**:这是核心价值主张——把自然语言意图转化为个性化优惠列表。没有它,产品就没有 MVP。

**独立可测试**:测试用户可以提交任何一种典型问法(仅品类、带商家名、带人数、带时间、带距离),无需任何 Agent 协同,即可在延迟预算内得到 ≥ 3 条相关项的个性化列表(可能混合四类内容形态)。

**验收场景**:
1. **假设** 已登录用户且在地理覆盖范围内,**当** 用户问"附近有什么咖啡优惠"且无其他约束,**则** 系统返回一组在用户合理距离内的内容列表(可能混合上述四类形态),按个性化相关性排序,并以标准商品卡(内容类型、名称、价格、折扣、距离、门店、有效期)展示。
2. **假设** 用户的提问带有明确约束(人数、时间段、距离、商家名),**当** 请求被处理,**则** 返回的每一条都满足全部已声明约束,且约束在商品卡上可见。
3. **假设** 用户的提问与本地优惠无关(如"今天天气怎么样"),**当** 请求被处理,**则** 系统返回一句礼貌的引导话术,提示用户用支持的方式重新描述,且不返回任何推荐列表。
4. **假设** 用户处于活跃会话且已有上文,**当** 用户发起"便宜点的"、"再远一点没关系"等微调,**则** 系统沿用上一轮的槽位值,只更新发生变化的项。
5. **假设** 在所有约束下没有候选,**当** 系统返回,**则** 用户得到最近似的匹配(或一个带"放宽条件"一键操作按钮的空状态),而不是原始错误。
6. **假设** 用户的提问同时匹配到多类内容形态(如同时有"美团门店券"和"自拓展门店券"),**当** 系统返回,**则** 列表中各类内容均被合理呈现,每条卡片上有清晰的内容类型标识(券/买单/外部券等),方便用户区分。
7. **假设** 用户的初始查询(如小众品类、冷门商圈)导致 ES 候选数低于召回阈值,**当** 系统处理,**则** 触发联想+语义召回(LLM 扩词 → ES 关键词+向量双路召回 → 去重),用更宽的相关语义补足候选;如果联想后仍无候选,才走空状态。**这一场景的边界:LLM 只生成相关词汇,不直接生成商品,所有商品必须来自 ES 召回结果**(防幻觉护栏)。

### 用户故事 2 —— Agent 协助下单(优先级:P2)

用户看完推荐列表后,通过对话告诉 Agent *"帮我下第 2 个"*,*"这家来 2 份"*,或 *"换第一家的券"* 来调整"购物车",最后说 *"确认下单并支付"*。Agent 负责准备订单、让用户调整数量与切换券/品,完成支付。

**为什么是这个优先级**:只做"发现"无法完成转化,下单 + 支付的闭环才是 GMV 的来源。之所以是 P2,是因为它依赖用户故事 1 的输出。

**独立可测试**:测试用户可以在纯对话(不必依赖按钮点击)的路径下完成 *发现 → Agent 准备订单 → 调整数量 → 确认 → 支付* 全流程,并最终看到订单确认与支付结果。

**验收场景**:
1. **假设** 当前已有一张推荐列表,**当** 用户说"下第 2 个"或"要第 2 个",**则** Agent 为该商品准备一份待支付订单草稿,并展示一张确认卡(名称、数量、价格、门店、"确认支付"操作按钮)。
2. **假设** 已有一份待支付订单草稿,**当** 用户说"改成 3 份"或"换成第 3 个券",**则** 草稿被原地更新,确认卡同步刷新。
3. **假设** 已有一份待支付订单草稿,**当** 用户说"确认支付",**则** 系统走标准 H5 支付流程,并以成功(带订单 id)或失败(带原因)的单条消息回复。
4. **假设** 支付失败(网络、限额、库存不足),**当** 失败被回传,**则** Agent 用人类可读的原因告知用户,并提供唯一一个下一步动作(重试 / 换一个 / 取消)。
5. **假设** 用户在 P1 还没产出列表时就说"帮我下单",**当** Agent 响应,**则** 它先做一次短澄清(或自动跑一次默认的发现),而不是凭空猜测商品。

### 用户故事 3 —— 对话式重新排序(优先级:P4)

已经有一张推荐列表的用户说 *"按距离排一下"*,*"按热度排"*,*"按价格排一下"*,或 *"综合排序回到默认"*,Agent 在不重新检索的前提下,对当前列表做重排并原地刷新。

**为什么是这个优先级**:排序是一个轻量控制,显著提升"被理解"的体感,也证明 Agent 真的把列表"握在手里"。之所以是 P4,是因为它是对 P1 的润色,而非新能力。

**独立可测试**:P1 已经返回列表后,用户可以发出任一支持的排序指令,可见列表按新顺序刷新,内容集合保持不变(没有新增/删除),并且响应文案反映了新的排序键。

**验收场景**:
1. **假设** 当前有一张推荐列表,**当** 用户说"按距离排一下"或"由近到远",**则** 列表按距离升序重排,响应文案中确认新的排序键。
2. **假设** 当前有一张推荐列表,**当** 用户说"按热度排",**则** 列表按目录热度信号降序重排,响应文案中确认新的排序键。
3. **假设** 用户发出排序指令,**当** 排序被应用,**则** 不会触发新的检索调用(列表内容不变),且排序结果在不做整页刷新的前提下可见。
4. **假设** 当前有一张推荐列表,**当** 用户说"按价格排一下"或"由便宜到贵",**则** 列表按价格升序(或用户指定的"由贵到便宜"对应降序)重排,响应文案中确认新的排序键。
5. **假设** 当前有一张推荐列表,**当** 用户说"推荐模型 / 综合 / 默认",**则** 列表回到推荐模型给出的默认排序(即不再按距离/热度/价格强制排序),响应文案中确认回到默认。

### 用户故事 4 —— 为未来"积分商城子 Agent"留出扩展点(优先级:P8,第二期)

主 Agent 在设计上预留接口,使第二期可以"即插即用"地接入积分商城子 Agent,无需重写路由层、会话状态或 H5 协议。从用户视角,他们可以问"我有多少积分"或"用积分换这个券",系统应能完成路由,以及(必要时)跨子 Agent 协同。

**为什么是这个优先级**:这是 P1~P7 上的一个非功能性设计约束,本身不是可交付功能。它出现在本规格中,是为了锁定接口边界,避免第一期把路走死。

**独立可测试**:在第一期主 Agent 跑起来但**未注册**积分商城子 Agent 时,问一个积分相关问题,得到一条干净的"暂未支持"兜底(而不是报错);接入一个桩子 Agent 后,同样的问题被正确路由,响应形态与本地优惠子 Agent 一致。

**验收场景**:
1. **假设** 积分商城子 Agent 尚未注册,**当** 用户问积分相关问题,**则** 主 Agent 用一句话告知该能力即将上线,既不崩溃也不误判。
2. **假设** 第二期积分商城子 Agent 已注册,**当** 用户问"我有多少积分"或"用积分换这个券",**则** 主 Agent 路由到该子 Agent,响应使用与本地优惠相同的 Envelope。
3. **假设** 出现跨垂直的查询(如"用积分换刚才那个咖啡券"),**当** 主 Agent 处理,**则** 它在单次用户轮次内串起"积分商城(余额校验)→ 本地优惠(券资格校验)",并以统一结果回报。

### 用户故事 5 —— 对话式排除与跨会话偏好记忆(优先级:P5)

用户在浏览推荐列表时,说 *"我不要瑞幸"*、*"这家以后都别推"*、*"咖啡我戒了"* 等表达,系统应当:立即把当前列表(或新一轮推荐)中的相关项过滤掉;然后**主动追问**用户是否要把这条偏好持久化下来(以后都不再推荐);若用户确认,则写入跨会话偏好存储,在后续所有轮次(不止本会话)中生效。

跨会话偏好同时支持**显式**与**隐式**两类:**显式**指用户主动说"我不要 / 我喜欢"产生的偏好(保留 180 天);**隐式**指从用户的点击 / 跳过 / 收藏 / 领取等行为信号推断出的偏好(保留 30 天)。**隐式偏好默认关闭**,需用户在 H5 设置里显式开启;一旦开启,系统只能记"统计信号"(如某品类的负向权重),不记"原始行为"。

**为什么是这个优先级**:把"用户明说不要" 沉淀为长期偏好,是让推荐"越用越懂你"的关键投入,显著提升后续轮次的命中率与用户体感。之所以是 P5,是因为它建立在 P1 之上且需要新基础设施(preference store),属于对 P1 的能力深化,而非 MVP 必需。

**独立可测试**:测试用户可以:
- 临时表达偏好(本次过滤,会话结束即丢);
- 表达偏好并确认持久化(后续会话里,该偏好项不再出现在结果中);
- 在持久化的偏好项上**临时解除一次**(例如"今天给我看看瑞幸吧"),之后偏好继续生效;
- 在 H5 设置里关闭隐式偏好后,系统不再记录新的隐式信号。

**验收场景**:
1. **假设** 当前有一张推荐列表,**当** 用户说"我不要瑞幸",**则** 当前列表立即过滤掉瑞幸相关项,主 Agent **主动追问**:"要把'不再推荐瑞幸'记下来,以后都看不到吗?",并提供"是 / 本次就好"两个按钮(可走 H5 actions 字段)。
2. **假设** 上一场景中用户选了"是",**当** 同一用户重新打开会话、再次搜索咖啡,**则** 瑞幸相关项**不出现**于新列表中。
3. **假设** 用户已经持久化过"不要瑞幸",**当** 他在新一轮里说"今天给我看看瑞幸",**则** 当前轮次临时显示瑞幸(覆盖持久化),但关闭该轮次后,持久化偏好继续生效。
4. **假设** 隐式偏好开关关闭(默认值),**当** 用户多次跳过某品类,**则** 系统**不**基于此行为生成新的隐式偏好。
5. **假设** 隐式偏好开关开启,**当** 用户多次跳过某品类,**则** 系统在后续推荐中降低该品类的权重;且偏好仅记"信号摘要"(品类 + 方向),**不**记原始点击 / 跳过日志。
6. **假设** 显式偏好的保留期(180 天)到期,**当** 该用户再次触发推荐,**则** 该偏好已自动失效,被排除项可能重新出现(若不再有其他约束)。

### 用户故事 6 —— 失败交易的归因与运营优化(优先级:P7)

当用户在一段会话中**未完成交易**(例如:草稿订单被取消、支付失败、列表中没有任何项被点过即关闭会话、用户主动放弃),系统应保留该会话的**完整交互回溯**数据,供运营 / 推荐 / 提示词团队后续离线分析:"这次推荐为什么没能促成交易?",用于优化推荐效果与 LLM 交互方式。

**为什么是这个优先级**:这是闭环优化的关键投入,能让"模型越用越好"有了数据基础。之所以是 P7,是因为它不直接面向最终用户,投入产出比需要等一段时间才能显现;但**架构上必须在第一期就预留**回溯管道,否则事后补齐成本极高。

**独立可测试**:运营/分析人员拿到一份"失败交易样本"(例如 100 条"用户走到了草稿订单但未支付"的会话),能逐条回看:用户最初问了什么 / 系统识别了哪些槽位 / 候选集里有什么 / 推荐返回了什么 / 用户在哪个环节犹豫或放弃 / 最后一句对话是什么;并能基于此提出"应当改进哪一环"的具体动作。

**验收场景**:
1. **假设** 一段会话最终未完成交易(以下任一:草稿被取消、支付失败、列表展示后无任何点击会话即结束、用户主动放弃),**当** 会话结束(或关键事件触发),**则** 系统写入一条"会话回溯记录",至少包含:用户初始查询、抽取的槽位、候选集大小、推荐返回的 top-K、用户对列表的每一步动作(看 / 排序 / 排除 / 下单 / 取消)、最终态(订单 id 或放弃原因)、关联 id。
2. **假设** 运营/分析人员拿到一批"失败交易"样本,**当** 其逐条回看,**则** 能定位到"在第几步 / 哪个提示 / 哪个候选上,用户行为偏离了'完成交易'路径",并据此提出具体优化项。
3. **假设** 一次交易**成功**完成,**当** 会话结束,**则** 同样写入回溯记录(可标记为"成功"),以建立"成功 vs 失败"对照样本。
4. **假设** 同一会话内用户切换了多个意图(如从"找咖啡"切到"查我的券包"),**当** 写入回溯,**则** 每段子意图都保留,完整还原"用户的目标漂移轨迹"。

### 用户故事 7 —— 券包管理(优先级:P3)

用户在与 Agent 的对话中可以查看自己**持有 / 已使用 / 已过期 / 还能使用**的券,做"我有哪些券可用 / 这张券还能用吗 / 我过期的券有哪些 / 我的券包" 等查询;返回的是 per-user 的券包视图(资产视角),而非新一轮"推荐"。

**为什么是这个优先级**:这是**售后对话的入口**,与"下单"形成完整闭环——用户买完之后自然会想"我有哪些券可以用 / 看看我的券包"。如果 Agent 不能回答,用户就会切到 H5 后端查,体验断成两截。之所以是 P3,是因为它依赖 P2(下单)产出的"已领券"数据,但相对独立、可独立交付。

**独立可测试**:测试用户可以:
- 查询"我的券包"(默认按某种合理顺序展示可用的券);
- 按状态过滤("我过期的券" / "我还能用的券");
- 点击某张券查看详情(规则、有效期、可用门店);
- 在多张券之间横向对比(详见 4. 验收场景 4)。

**验收场景**:
1. **假设** 用户当前没有任何券,**当** 用户说"看看我的券包"或"我有哪些券",**则** 系统返回空状态提示("您还没有领过券,要不要现在看看有什么优惠?"),而非错误。
2. **假设** 用户当前持有多种状态的券(可用 / 已用 / 过期),**当** 用户说"我有哪些券",**则** 系统按默认顺序(可用优先 → 已用 → 过期)展示券包,并在 UI 上以状态标签(可用 / 已用 / 过期)分组。
3. **假设** 用户想看某张券的详情,**当** 用户说"看看第 2 张券"或问"这张券还能用吗",**则** Agent 返回该券的详情(规则、有效期、可用门店、是否已领完/已用完),并以一张"券详情卡"展示。
4. **假设** 用户想在多张可用券之间对比,**当** 用户说"对比第 1 张和第 3 张",**则** Agent 返回一张横向对比卡(价格、折扣、有效期、可用门店集合),帮助用户做选择。
5. **假设** 用户的某张券**即将过期**(剩余有效期 ≤ 7 天),**当** 用户查询券包,**则** 该券以"快过期"角标展示;若用户在本轮内说"用这张",Agent 引导进入下单流程(交给 LP 的下单工具)。
6. **假设** 用户查询"我还能用的券"(显式过滤状态 = 可用),**当** 系统返回,**则** 列表中**不**包含已用、已过期、已被领完的券,且每张券都附带剩余有效期。

### 用户故事 8 —— 上下文压缩(优先级:P6)

当一段会话持续很久(轮次或累计字符超过阈值)时,主 Agent 应当自动把"较早"的轮次压缩为一段**摘要**,只把"最近 N 轮"原样保留——这样后续每轮 LLM 调用收到的上下文依然有界、可控,长会话不会因上下文爆炸而掉质量或掉成本。

压缩后的会话状态由两部分组成:
- **摘要**(1 段,LLM 生成的总结):保留用户核心意图的演变、关键槽位的变化、已展示列表的摘要、已发生的关键动作(下单 / 改单 / 排除 / 排序)、持久化偏好项。
- **最近轮次原样**:最近 M 轮对话 + 工具调用结果原样保留(不压缩)。

**为什么是这个优先级**:这是"主 Agent 自身能力"的非功能性优化,直接决定了**长会话的可用性**——一个 30 轮以上的会话如果不压缩,要么超过 LLM 上下文窗口,要么成本失控,要么 LLM 早期信息被截断。之所以是 P6,是因为 MVP(典型 5~10 轮)用不到;但**架构上必须在第一期就预留**,否则后期改造要重排主 Agent 状态机。

**独立可测试**:模拟一段 ≥ 30 轮的长会话,验证:
- 早期轮次被压缩为一段摘要,关键信息(槽位、列表、动作)未丢;
- 最近 M 轮原样保留;
- 后续 LLM 调用的输入 token 数稳定在阈值之下;
- 用户在第 30 轮仍能正确引用"我在第 5 轮说的那个咖啡店"——证明摘要保留了必要上下文。

**验收场景**:
1. **假设** 单次会话的轮次计数或累计输入字符超过压缩阈值(可配置,默认 20 轮或 8000 字符,先到为准),**当** 下一轮 LLM 调用发生,**则** 主 Agent **必须**先把"超阈值"部分压缩为一段摘要,与最近 N 轮原样拼接后,再送入 LLM。
2. **假设** 已发生压缩,**当** 用户在后续轮次引用"我之前说的 X",**则** Agent **必须**仍能从摘要中正确还原 X(摘要中已保留用户关键意图与关键槽位)。
3. **假设** 压缩后,**当** 主 Agent 渲染响应,**则** **不得**对用户暴露"我刚才压缩了上下文"之类的元信息;压缩对用户透明。
4. **假设** 压缩失败(摘要服务异常),**当** 下一轮 LLM 调用发生,**则** 主 Agent **必须**降级为"仅保留最近 N 轮 + 丢早期轮次",**不得**阻塞用户响应;失败信息进入会话回溯记录。
5. **假设** 摘要内容**不得**包含 PII(原始用户输入中的身份证号 / 手机号 / 银行卡号等),**当** 摘要生成,**则** 必须经过去标识处理(具体规则在 A-016 中约束)。

### 边界场景

- **范围外意图**:与优惠/下单/积分无关的提问(天气、闲聊、辱骂)。以优雅的引导回应,**绝不**返回原始错误。
- **无候选场景**:约束组合导致无匹配。展示带"放宽条件"一键操作按钮的空状态,而非空白列表。
- **位置缺失**:用户未授予位置信息。回退到用户档案中的"家地址",并明确标注"基于您的家地址"。
- **券过期/被领完**:用户从列表里选了某张券,到他说"下这个"时已不可领。走一条友好的"重新选择"流程,而不是直接报支付错误。
- **流程中切换意图**:用户在同一轮内从"找门店"切到"支付上一单"。主 Agent 结束推荐轮,再起订单轮,且不丢失上一轮的槽位。
- **空/模糊的商家名**:"我想去那家星巴克"但没说哪一家。Agent 只问一个消解问题,不要问五个。
- **候选爆炸**:目录返回 > 50 条候选。Agent 在送入排序器前必须硬性封顶(例如 20),以保证 Prompt 有界与成本可控。
- **中英混排输入**:用户混用中文与品牌名(如"瑞幸 luckin")。槽位抽取必须对混合语种 token 具备容忍度。
- **营业时间不匹配**:门店在距离内但请求时段关门。要么过滤掉,要么露出"当前歇业"角标。
- **会话续接**:用户几小时后再回来。会话上下文已过期,Agent 必须让用户重新确认约束,而非静默重跑。

## 需求 *(必填)*

### 功能性需求

#### 路由与会话(主 Agent)

- **FR-001(主)**:系统**必须**在响应前对每一轮用户输入做意图分类,落到支持的垂直意图之一(当前:`local_promotion` / `order` / `coupon_wallet`、`points_mall`、`out_of_scope`)。其中 `local_promotion` / `order` / `coupon_wallet` 都路由到同一个本地优惠子 Agent(它的工具集覆盖"发现"、"下单/改单/支付"与"券包管理"三类)。
- **FR-002(主)**:系统**必须**对每个垂直意图把请求路由到**恰好一个**子 Agent;若该意图没有已注册的子 Agent,主 Agent **必须**返回一句"暂未支持"兜底,**不得**调用任何子 Agent。
- **FR-003(主)**:系统**必须**为每个登录用户维护**一份**会话上下文(当前列表、当前草稿订单、最近槽位、最近排序键、本次临时排除项、持久化偏好的临时覆盖标记),**跨多端(H5 在手机 / PC / 平板同时打开)共享同一份**——任何一端的列表变更、排序、改单 / 下单操作**必须**同步刷新到其他端;并**必须**按用户隔离;**禁止**跨用户串数据。
- **FR-004(主)**:当一轮用户输入跨多个垂直时(如"用积分换这个券"→ 积分商城 + 本地优惠),系统**必须**支持单轮内跨子 Agent 协同,并以单一统一响应回报。
- **FR-005(主)**:系统**必须**对外发出一条连贯的自然语言回复,附带结构化 payload;**不得**向用户泄漏内部工具名、槽位名、子 Agent 标识。

#### 本地优惠子 Agent(第一期)

- **FR-010(本地优惠)**:子 Agent **必须**从用户提问中抽取如下槽位(在用户提到时):`category`(品类)、`name`(商家名称)、`number`(套餐人数)、`time`(营业时间)、`distance`(距离);允许缺失。
- **FR-011(本地优惠)**:子 Agent **必须**仅在用户提问中至少出现 {品类、商家名、人数、时间、距离、折扣关键词、门店关键词} 之一时,才判定为范围内;否则**必须**向主 Agent 返回 `out_of_scope`。
- **FR-012(本地优惠)**:子 Agent **必须**使用商品名称检索与距离检索能力,从美团门店券、自拓展门店券、本地优惠买单、外部券四类内容中拉取候选集,并对候选集大小做硬性封顶,避免排序器输入爆炸。
- **FR-012b(ES 降级)**:当 ES 调用异常 / 超时时,子 Agent **必须**回退到"最近的缓存候选集"(缓存由推荐服务侧维护,缓存键含用户位置 / 品类 / 时间),并在响应中标注"已启用降级,可能不是最新结果";若缓存也无可用候选,才走空状态。ES 异常**必须**触发告警(由 FR-051 关联 id 追踪)。
- **FR-013(本地优惠)**:子 Agent **必须**在排序前拉取用户画像与最近行为序列,并把它们作为入参传给推荐服务。
- **FR-014(本地优惠)**:子 Agent **必须**调用推荐服务(候选集 + 用户上下文),请求 top-K 结果(K 可配置,默认 10)。
- **FR-014b(推荐服务降级)**:当推荐服务异常 / 超时时,子 Agent **必须**跳过 LLM 排序,直接以候选集 + 热度信号做客户端规则排序,并在响应中标注"已启用降级,推荐精度下降";**若 ES 也同时异常**(两路全挂),返回友好错误 + "请稍后重试"入口,绝不返回空列表伪装成正常结果。降级路径**必须**经过冒烟测试覆盖。
- **FR-014c(新用户回退)**:当用户被识别为"新用户"(无画像或无行为序列)时,子 Agent **必须**走"全站热度榜"回退:跳过 LLM 个性化排序器,以候选集 + 热度信号做客户端规则排序;**响应中不**带"新用户推荐"角标(用户体感与老用户一致);待用户行为累积(由 plan 阶段定阈值,默认 5 次浏览或 1 次下单)后再切换到个性化排序。
- **FR-015(本地优惠)**:子 Agent **必须**对 top-K 中每一条做券门店资格与单人限额的补全,并**必须**把内部券与对应门店合并。
- **FR-016(本地优惠)**:子 Agent **必须**返回结构化结果:推荐项列表、实际生效的槽位、当前排序键、面向用户的一句话摘要。
- **FR-017(联想+语义召回)**:当 ES 候选集低于召回阈值(可配置,默认 5)时,子 Agent **必须**触发"联想+语义召回"流程:用 LLM 基于用户原始 query 生成相关扩展词汇(只产词,不直接产商品)→ ES 用这些扩展词做**关键词 + 向量双路召回**→ 合并两路结果并**去重**(以商品 id 为键)→ 再走常规的画像/排序/补全链路。**幻觉护栏:任何最终出现在结果里的商品,必须来自 ES 的召回结果,不得直接由 LLM 生成。**
- **FR-018(对话式排除)**:子 Agent **必须**提供 `exclude_items` 工具,支持按商品/品牌/品类关键词对当前列表(或新一轮推荐)进行过滤;**默认**只作用于当前结果(会话级临时排除),**不**自动持久化。在过滤生效后,主 Agent **必须**主动追问用户:"要把'不再推荐 X'记下来,以后都看不到吗?",并以 H5 `actions` 字段提供"是 / 本次就好"两个按钮;用户选择后才决定是否写入持久偏好。
- **FR-019(跨会话记忆)**:系统**必须**维护一个 per-user 的 preference store,记录两类偏好:
  - **显式偏好**(用户主动说"我不要 X / 我喜欢 Y"):保留期 180 天,自动过期。
  - **隐式偏好**(从用户行为信号推断):保留期 30 天,自动过期;**默认关闭**,需用户在 H5 设置中显式开启;开启后也**只记统计信号摘要**(如"品类 X 负向权重 ↑"),**不记原始行为日志**。
  - 偏好**必须**在 LP 子 Agent 拉取用户画像/排序前生效(对检索/排序两层都生效);允许"临时覆盖一次"(如已持久化的偏好项,在用户当场说"今天给我看看"时,本次轮次生效,关闭后恢复持久化状态)。
- **FR-019b(数据主体权利 · 删除)**:系统**必须**支持用户"删除我的偏好":H5 设置里提供"立即清除"按钮,点击后该用户**所有**显式偏好**必须**在 1 分钟内物理删除(从 preference store 抹除,不留软删除);隐式偏好继续走自动过期;**正在生效的"临时覆盖"标记也一并清除**。**不得**清除用户画像与行为序列(这些由 H5 用户中心统一处理,不归 Agent);**不得**清除历史订单(由订单服务持有)。删除操作**必须**写入 Conversation Trace(脱敏后)以便审计。

#### 下单能力(作为本地优惠子 Agent 内部的工具集合)

- **FR-020(下单工具)**:本地优惠子 Agent **必须**提供工具,支持从列表序号引用("第 2 个")、商家名引用("那家星巴克")、商品 id 引用三者中任一种准备订单草稿;在存在歧义时**必须**最多只追问一次。
- **FR-021(下单工具)**:本地优惠子 Agent **必须**提供工具,支持对草稿订单的对话式修改:改数量、换商品、换券、取消、进入支付。
- **FR-022(下单工具)**:本地优惠子 Agent **必须**把支付交回 H5 标准支付流程,并以一条聊天消息 + 结构化 payload 回报结果(成功带订单 id,失败带原因)。
- **FR-023(下单工具)**:本地优惠子 Agent **必须**在支付前对商品、数量、券资格做一次最终校验;若已失效(售罄、券已被领完等),走一条友好的"重新选择"流程。
- **FR-023b(下单互斥,多端)**:主 Agent **必须**在支付前对草稿订单做**互斥校验**:同一 user 在多端只能有一份"正在支付"的草稿;若检测到另一端正在支付(状态 = `awaiting_payment` 且未超时),**必须**拒绝本次支付并提示用户"另一端正在支付,请稍候或先去那里完成"。互斥校验**不得**阻塞主流程超过 200ms。

#### 券包管理(作为本地优惠子 Agent 内部的工具集合)

- **FR-024(券包工具)**:本地优惠子 Agent **必须**提供工具,支持用户查询自己**持有 / 已使用 / 已过期 / 还能使用**的券,以及查看某张券的详情 / 在多张券之间做对比。具体:
  - `list_my_coupons(status?, sort?, filters?)`:按状态(可用 / 已用 / 过期)、排序(过期时间 / 面值 / 门店)、其他过滤条件返回券列表;无券时返回空状态。
  - `get_coupon_detail(coupon_id)`:返回单张券的规则、有效期、可用门店、是否已领完 / 已用完。
  - `compare_coupons(coupon_ids[])`:横向对比多张券(价格、折扣、有效期、可用门店集合)。
- **FR-025(券包与下单衔接)**:在券包查询结果中,若用户对某张券表达"用这张 / 下这个",**必须**走 LP 内部的下单工具(FR-020)准备草稿订单,**不得**绕过下单工具直接调收银台。
- **FR-026(快过期引导)**:若某张券剩余有效期 ≤ 7 天,**必须**在卡片上露出"快过期"角标;若用户在本轮内说"用这张",Agent 引导进入下单流程(交给 LP 的下单工具)。

#### 排序与列表控制(P3)

- **FR-030(列表)**:系统**必须**支持以下**对话式排序指令**作用于当前列表,且不重新检索:**`distance`(按距离 / 距离最近)、`popularity`(按热度 / 热门优先)、`price`(按价格,可指定升序或降序)、`model`(推荐模型默认 / 综合排序)**。未知排序短语**必须**回退到当前排序。**第一期不支持按有效期排序**(已确认推迟到第二期积分商城再讨论)。
- **FR-031(列表)**:排序指令**不得**改变列表的内容集合,只改变顺序;**必须**在响应文案与结构化 payload 的 `sort_key` 字段中体现,且响应文案明确告知用户当前的排序键。

#### 可扩展性(第二期,只锁契约)

- **FR-040(扩展)**:主 Agent **必须**把子 Agent 集合视为一份注册表;新增子 Agent **不得**要求修改主 Agent 的路由逻辑、会话状态代码或 H5 协议。
- **FR-041(扩展)**:每个子 Agent **必须**实现统一的响应 Envelope `{items?, summary, slots_used, sort_key?, actions?, follow_up?}`,使主 Agent 能以统一方式渲染。
- **FR-042(扩展)**:主 Agent **必须**对未注册垂直提供"暂未支持"兜底。

#### 非功能性

- **FR-050(NFR)**:典型推荐轮次的端到端响应延迟必须有界(发现路径 p95 目标 ≤ 4 秒)。
- **FR-051(NFR)**:每一轮用户输入**必须**可端到端追踪,日志中带同一关联 id(且**不得**包含 PII)。
- **FR-052(NFR)**:主 Agent 与每个子 Agent **必须**可独立部署、独立版本化、独立回滚。
- **FR-053(失败交易回溯)**:系统**必须**为每一段会话写入一条"会话回溯记录"(conversation trace),覆盖以下任一最终态:交易成功 / 草稿被取消 / 支付失败 / 列表展示后无任何点击会话即结束 / 用户主动放弃。记录**必须**包含:关联 id、用户初始 query、抽取的槽位、ES 候选集大小、联想召回是否触发、推荐返回的 top-K、用户对列表的每一步动作(看 / 排序 / 排除 / 下单 / 改单 / 取消)、最终态(订单 id 或放弃原因)、同一会话内的子意图切换轨迹。记录的写入是**异步、不阻塞**主流程的;写入失败**不得**影响用户响应。回溯记录的消费者是离线分析与运营团队(不是实时系统),**因此只记用于离线分析所需的最小字段,且不得包含 PII**。
- **FR-054(上下文压缩)**:主 Agent **必须**监控每段会话的轮次计数与累计输入字符数;当任一指标**超过压缩阈值**(可配置,默认 20 轮或 8000 字符,先到为准)时,**必须**触发压缩:把"超阈值"的早期轮次由 LLM 压缩为一段摘要,与最近 N 轮原样拼接后,再送入下一轮 LLM 调用。摘要**必须**保留:用户核心意图演变、关键槽位变化、已展示列表的摘要、已发生的关键动作(下单 / 改单 / 排除 / 排序)、持久化偏好项;**不得**包含 PII(由 A-016 约束)。当压缩失败时,主 Agent **必须**降级为"仅保留最近 N 轮 + 丢早期轮次",**不得**阻塞用户响应;失败信息进入会话回溯记录(FR-053)。压缩对用户透明——**不得**对用户暴露"我刚才压缩了上下文"之类的元信息。
- **FR-055(per-user 限流)**:主 Agent **必须**对每个登录用户实施**限流**:每分钟最多 50 次输入;下单 / 改单 / 排除等"敏感操作"可由 plan 阶段进一步细化更严格的速率(默认 10 次/分钟)。**超限响应**:主 Agent 立即返回"操作太频繁,请稍候再试"提示,**绝不**静默丢消息、不延迟响应。限流计数按 user id 持有(不按 session id),多端共享同一份计数(与 A-017 一致);**不得**因为限流而阻断正常用户的连续交互(典型用户在 1 分钟内不会超过 10 次输入)。

### 关键实体

- **User Query(用户输入)**:一轮用户发言。属性:原文、session id、user id、时间戳、位置(若有)、设备语言。
- **Slots(槽位)**:从用户提问中抽出的结构化约束。属性:`category`、`name`、`number`、`time`、`distance`,以及各垂直私有的槽位。Slots 是主 Agent 与子 Agent 之间的契约。
- **Recommendation Item(推荐项)**:结果列表中的一条。属性:id、`content_type`(`meituan_coupon` | `self_operated_coupon` | `discount_checkout` | `external_coupon`)、主标题、副标题、图片、价格、折扣、绑定门店、距离、有效期、深链。其中 `discount_checkout` 表示"本地优惠买单"类内容(到店出示买单),其余三类表示不同来源的"券"。
- **Recommendation List(推荐列表)**:按 session 作用域有序的推荐项集合,带当前 `sort_key`。
- **Coupon(券)**:可领取的优惠。属性:id、标题、绑定门店集合、资格规则、单人限额、有效期、领取状态。
- **Owned Coupon(用户已持有的券)**:用户视角的"我的券包"中的一项。属性:id、coupon id(对应上面的 Coupon)、user id、状态(`available` | `used` | `expired` | `claimed_but_unavailable`)、领取时间、使用时间(若已用)、过期时间、绑定订单 id(若已用)、剩余有效期天数。数据源是用户已领券 / 已使用券的查询服务,Agent 调用,不拥有数据。
- **Store(门店)**:商家。属性:id、名称、位置、营业时间、品类集合、距用户距离。
- **Draft Order(草稿订单)**:用户正在搭建的订单。属性:items(带数量)、使用的券、总价、状态(`draft` | `awaiting_payment` | `paid` | `cancelled`)。
- **Payment Result(支付结果)**:一次支付尝试的产出。属性:order id、状态(`success` | `failed` | `cancelled`)、失败原因(若有)。
- **User Profile(用户画像)**:长期属性。属性:id、家地址、偏好品类、年龄段、语言。
- **User Behaviour Sequence(用户行为序列)**:按时间排列的近期行为(搜索、浏览、领券、购买),用于排序上下文。
- **Preference(偏好项)**:per-user 的一条偏好。属性:user id、目标(品牌/品类/商品 id)、方向(`dislike` | `like`)、来源(`explicit` | `implicit`)、置信度(仅隐式)、创建时间、过期时间。显式偏好保留 180 天,隐式偏好保留 30 天;过期自动失效。
- **Preference Store(偏好存储)**:跨会话的 per-user 偏好服务,提供写入、读取、按时间过期、临时覆盖等能力。
- **Conversation Trace(会话回溯记录)**:一段完整会话的离线分析样本。属性:关联 id、user id(脱敏)、起止时间、初始 query、槽位抽取轨迹、候选集大小、是否触发联想召回、推荐 top-K、用户对列表的每一步动作、最终态(成功订单 id / 草稿取消 / 支付失败原因 / 放弃原因)、同一会话内的子意图切换轨迹、是否触发上下文压缩(及压缩失败信息)。**不**包含 PII。
- **Session Summary(会话摘要)**:由上下文压缩(FR-054)生成的一段 LLM 摘要,代替超阈值的早期轮次送入后续 LLM 调用。属性:本段覆盖的轮次范围、用户核心意图演变、关键槽位变化、已展示列表的摘要、已发生的关键动作(下单 / 改单 / 排除 / 排序)、持久化偏好项。**不**包含 PII(由 A-016 约束)。
- **Sub-Agent Registry(子 Agent 注册表)**:把垂直意图映射到子 Agent 端点/版本的配置,主 Agent 启动时读取。第一期:本地优惠子 Agent(`local_promotion` + `order` + `coupon_wallet` 三个意图共享);第二期:新增积分商城子 Agent(`points_mall`)。

## 成功标准 *(必填)*

### 可度量结果

- **SC-001(发现)**:新用户提交任意范围内的查询,p95 在 4 秒内拿到个性化推荐列表。
- **SC-002(发现)**:≥ 90% 的范围内查询至少返回 1 条;带显式约束的查询中,≥ 95% 的返回项满足全部已声明约束。
- **SC-003(意图)**:在 `local_promotion` 与 `out_of_scope` 的二分类边界上,意图分类精确率达到 ≥ 95%(在带标签的评测集上衡量)。
- **SC-004(下单)**:用户可以在 ≤ 5 轮对话内完成"发现 → 调整 → 支付"全流程,且至少 80% 的会话达成。
- **SC-005(排序)**:每一条支持的对话式排序指令(`distance` / `popularity` / `price` / `model`)产出的列表,与上一轮相比**内容集合完全一致**、**顺序**与请求的排序键一致;以确定性排序键校验为准。
- **SC-006(扩展)**:向运行中的系统新增一个子 Agent(如积分商城),**必须**通过配置/注册即可完成,不得修改主 Agent 代码、会话状态代码、H5 协议;以"接入桩子 Agent 后所有第一期测试全绿"作为回归门。
- **SC-007(安全)**:100% 的 `out_of_scope` 查询都得到一句优雅的引导,绝不出现原始错误或推荐列表。
- **SC-008(成本)**:典型推荐轮次最多触发 2 次模型调用(一次路由/意图,一次排序),单轮成本有界且可预测。
- **SC-009(隔离)**:用户会话状态完全隔离;跨用户串数据被禁止,并**必须**被一个"交换 session id 后断言无重叠"的自动化测试捕获。

## 假设

- **A-001**:H5 前端已存在,并暴露一个支持流式响应、操作按钮、支付交接的聊天界面;本次只设计 Agent 侧。
- **A-002**:推荐服务由大模型平台以托管 API 形式提供,接受"候选集 + 用户上下文",返回 top-K;我们不拥有模型本身。
- **A-003**:商品名称与距离检索由现有 Elasticsearch(ES)集群提供,索引为标准的商品/门店索引;Agent 不拥有索引。
- **A-004**:用户画像与行为序列以查询接口形式由现有服务提供;Agent 调用,不拥有数据。
- **A-005**:券资格、单人限额、券门店绑定能力由现成服务提供;本地优惠子 Agent 调用,不拥有规则。
- **A-006**:本地优惠子 Agent 的支付由 H5 标准收银台流程完成,Agent 走交接而非自实现支付;第二期积分商城若涉及"兑换下单",走积分扣减而非收银台,这条链路与 A-006 无关,届时由积分商城子 Agent 自行处理。
- **A-007**:用户是已登录的 H5 用户,带 session id;Agent 不负责登录。
- **A-008**:第一期不包含积分商城子 Agent;本规格只锁定主 Agent 的契约,确保第二期是"纯新增"。
- **A-009**:"个性化"排序依赖用户画像与行为可用;**新用户(无画像 / 无行为序列)走"全站热度榜"——但**响应中**不**带任何"新用户推荐"角标**,**用户体感与老用户一致**;排序仍走正常的"ES 候选 + 热度排序器"路径,跳过 LLM 个性化排序器;待用户行为累积后再开启个性化(具体阈值由 plan 阶段定,默认 5 次浏览或 1 次下单后开启)。
- **A-010**:延迟预算假设候选集检索是主要耗时,排序器按"单次有界调用"处理。
- **A-011(跨会话记忆)**:Preference Store 由后端服务提供;显式偏好保留 180 天,隐式偏好保留 30 天,过期自动失效;数据脱敏(只存"目标 + 方向 + 来源",不存原始用户输入);用户可在 H5 设置里关闭隐式偏好与查看/删除显式偏好。
- **A-012(隐式偏好默认关闭)**:第一期上线时,隐式偏好采集**默认关闭**;用户在 H5 设置中显式开启后才记录隐式信号。**这是隐私与体验的折中:不替用户做决定,也不强迫用户做选择**。
- **A-013(联想召回护栏)**:LLM 在联想召回中**只生成相关词汇**,不直接生成商品;所有最终出现的商品必须来自 ES 关键词+向量双路召回的合并去重结果;此为防幻觉的硬约束。
- **A-014(失败交易回溯)**:Conversation Trace 的写入是**异步且不阻塞主流程**的;写入失败不得影响用户响应;Trace 的消费者是离线运营/分析,不是实时系统,故只存"用于离线分析的最小字段",不存 PII。
- **A-015(券包数据源)**:用户已持有的券(可用 / 已用 / 过期)由后端"用户资产查询服务"提供,Agent 通过该服务查询,**不**自己持有数据;券详情 / 券可用门店 / 券规则等字段由券服务提供,Agent 调用。
- **A-016(上下文压缩阈值与隐私)**:压缩阈值(轮次 / 字符)可配置,默认 20 轮或 8000 字符(先到为准);摘要生成**必须**经过去标识处理,不写入身份证号 / 手机号 / 银行卡号等 PII;压缩失败降级策略在 FR-054 中锁定。
- **A-017(多端共享同一份会话)**:同一 user 在多端打开 H5 时共享一份 session state(由主 Agent 持有),不区分端;**第一期不做"踢掉旧 session"机制**;两端间的状态同步通过 H5 推送或下轮 LLM 调用前主动拉取实现(具体协议在 plan 中落地)。
- **A-018(降级阶梯的 SLO 假设)**:ES 的可用性目标 ≥ 99.5%,推荐服务的可用性目标 ≥ 99%(具体数字由各服务的 SLO 决定);**当任一服务在连续 1 分钟内失败率 > 30% 视为降级触发条件**;降级路径的延迟上限:ES 缓存回退 ≤ 200ms,推荐服务跳过 LLM 排序 ≤ 500ms;具体的告警阈值与切换延迟在 plan / 运维文档中落地。
- **A-019(数据保留与合规)**:第一期不主动支持跨境数据传输;所有用户数据(偏好、行为、券包、订单、Conversation Trace)按 PIPL 要求保留;回溯记录按**聚合 + 脱敏**保留 N 个月(可分析,不可定位个人,默认 6 个月,具体由合规与运营协商);用户在 H5 设置中可发起"删除我的偏好"操作(FR-019b);**导出我的数据**作为第二期或用户中心的能力,**第一期不锁**。

## 设计——主/子 Agent 功能划分

本节锁定 WHAT(以及最低限度的 WHY);具体的 HOW(工具、提示词、schema)放在 plan 中。

### 主 Agent(路由/编排者)

**职责**:
- 用户面对面的会话状态(槽位、当前列表、当前草稿订单、当前排序键、本次临时排除项),按 session 持有。
- 意图分类与基于注册表的子 Agent 路由。
- 单轮跨垂直协同(第二期及以后)。
- 响应组装:把子 Agent 的结构化 Envelope 合并成"一条聊天消息 + 结构化 payload"给 H5。
- 优雅兜底:`out_of_scope` 引导、垂直未注册时的"暂未支持"、空状态操作。
- **排除商品的"持久化追问"对话流**:在 LP 过滤生效后,主动向用户询问"是否要记下来,以后都不再推荐",并以 H5 `actions` 字段提供"是 / 本次就好"按钮;用户选择后才决定是否写入 Preference Store。
- **会话结束/关键事件时,异步触发 Conversation Trace 写入**(不阻塞主流程)。
- **上下文压缩(FR-054)**:监控会话长度,触发后把早期轮次压缩为 Session Summary,与最近 N 轮拼接后喂给下一轮 LLM 调用;压缩对用户透明;失败降级为"丢早期轮次",不阻塞响应。

**非职责**:
- 不做检索、排序、券规则、支付等任何垂直特定逻辑。
- 不持有长期用户数据(只调用用户画像/行为服务、Preference Store 服务)。

### 子 Agent 1——本地优惠(第一期,核心)

**职责**:
- 把路由过来的问句精炼为结构化槽位(`category`、`name`、`number`、`time`、`distance`)。
- 通过 ES 做候选集检索(商品名 + 距离);**候选不足时触发联想+语义召回**(LLM 扩词 → ES 关键词+向量双路召回 → 去重)。
- 调用用户画像与行为序列服务。
- 读取 Preference Store 中的**有效**显式 / 隐式偏好,并在检索与排序前应用(检索层排除、排序层降权)。
- 用"候选集 + 用户上下文"调用推荐服务,得到 top-K。
- 对 top-K 做券门店资格、单人限额补全,并把内部券与门店合并。
- 支持 `exclude_items` 工具,按品牌/品类/商品 id 对当前列表或新一轮结果做过滤;默认只作用于本次结果,不自动持久化(由主 Agent 走"持久化追问"流程)。
- 以统一 Envelope 返回结果。

**非职责**:
- 不持有会话状态(每次调用都是无状态的,会话状态由主 Agent 持有)。
- 不做支付。
- 不在"已返回结果后"做排序(主 Agent 持有列表,负责对话式重排)。
- 不做跨会话偏好持久化(只读 Preference Store,写操作由主 Agent 触发)。

### 子 Agent 1 内部工具分层

本地优惠子 Agent 内部的工具集分为**四层**(便于第二期做边界判断):

- **推荐工具**:候选集检索(ES,含联想召回路径)、用户画像/行为调用、Preference 读取、推荐服务调用(top-K)、券门店资格/限额补全、内部券与门店合并。
- **下单工具**:草稿订单准备、对话式改单、支付前最终校验、H5 支付交接与结果回报。
- **券包管理工具**:`list_my_coupons`、`get_coupon_detail`、`compare_coupons`;数据由后端"用户资产查询服务"提供,Agent 调用,不拥有数据。
- **排除与偏好工具**(只读 + 临时过滤):`exclude_items`(作用于本次结果,不持久化);Preference 的**写**由主 Agent 负责(走"持久化追问"流程),不放在这里。

四层都从属于"本地优惠"这一垂直,因此不引入额外的 Agent 边界;但分层使第二期(若积分商城需要下单/偏好/券包视图)可以基于这一边界判断"抽离/共享/复制"。

### 子 Agent 2——积分商城(第二期桩)

第一期不实现。在注册表里留空槽,使主 Agent 的路由处理"暂未支持"不需要任何代码改动。

**第二期接入时的方向性提示(不作为第一期承诺)**:
- 内容形态:积分商城实物/虚拟商品。
- 支付模型:**积分扣减**(消耗用户既有积分,无收银台接入)。
- 兑换链路:与本地优惠的"收银台 + 券核销"链路**完全不同**,不与本地优惠的下单工具共享实现。
- 跨垂直场景:"用积分换刚才那个咖啡券" 这种查询,需要积分商城(余额/兑换规则)与本地优惠(券资格)的协同,主 Agent 的"单轮内跨子 Agent 协同"能力(FR-004)就是为这种场景预留的。

## 设计——交互方式

### 会话 Envelope(每个子 Agent → 主 Agent)

每个子 Agent 都返回统一 Envelope,主 Agent 以统一方式渲染:

| 字段          | 类型                    | 作用                                                   |
|---------------|-------------------------|--------------------------------------------------------|
| `items`       | `RecommendationItem[]`? | 结果列表(若有)                                          |
| `summary`     | string                  | 面向用户的一句话(已经使用用户的语言)                       |
| `slots_used`  | `Slots`                 | 实际生效的槽位(用于透明化展示与追问)                      |
| `sort_key`    | string?                 | 当前生效的排序键                                        |
| `actions`     | `Action[]`?             | 建议展示在 H5 上的下一步操作按钮                          |
| `follow_up`   | string?                 | 一个可选的追问话术,主 Agent 可原样回传给用户               |

这个 Envelope 是"新增子 Agent = 纯注册"这一点的契约。

### H5 ↔ 主 Agent

- **入站**:文本(打字)或语音(已转写为文本),附带 session id 与 user id。
- **出站**:流式文本 + 一份结构化 payload(items / actions / follow_up)。H5 由同一份 payload 渲染聊天气泡、商品卡、操作按钮;主 Agent **绝不**把单次响应拆成多个聊天气泡。

### 主 Agent ↔ 子 Agent

- **调用风格**:结构化请求/响应(类似工具调用)。主 Agent 把"子 Agent 可见的会话上下文"(当前槽位、当前列表摘要等)打包下发,子 Agent 返回 Envelope。
- **无状态**:子 Agent 跨调用无状态。所有会话状态由主 Agent 持有——这是"子 Agent 可独立部署"的前提。
- **单轮串联**:主 Agent 可以在同一用户轮次内调用多个子 Agent(第二期:例如积分 + 本地优惠),并合并多份 Envelope。

### H5 ↔ H5 后端(不在 Agent 范围)

- 领券、展码、核销由 H5 后端持有;Agent 在"领券"操作处把手交给 H5 后端,不再参与。
- 支付是 H5 标准支付流;本地优惠子 Agent 的下单工具在"确认支付"操作处把手交给 H5。

### 重排与列表修改

- 主 Agent 在会话状态中持有当前列表与 `sort_key`。对话式排序指令更新 `sort_key` 并原地刷新列表;排序过程不调用任何子 Agent。
- 单条级别的修改(如"下第 2 个"、"改成 3 份")由主 Agent 路由到本地优惠子 Agent(因为下单工具在该子 Agent 内部),并附带"指向当前列表中某条"的引用。由于下单与发现在同一子 Agent 内,上下文无需跨 Agent 序列化,延迟最低。
- **排除商品的修改**(如"我不要瑞幸")走 LP 的 `exclude_items` 工具,默认只过滤当前结果;主 Agent 在过滤生效后主动追问是否持久化(详见 FR-018 与用户故事 5)。
